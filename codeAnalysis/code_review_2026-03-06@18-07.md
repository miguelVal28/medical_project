## Security Vulnerabilities

### SEC-01: PostgREST Filter Injection in Patient Lookup [CRITICAL]

**File:** `src/services/patientsService.ts:13`

```typescript
.or(`email.eq.${email}, document.eq.${document}`)
```

The `email` and `document` values are directly interpolated into the Supabase PostgREST filter string without sanitization. An attacker could craft input like `foo,id.gt.0` to inject arbitrary filter conditions, potentially returning all patient records.

**Impact:** Full patient data exposure. An authenticated user could extract the entire patient database.

**Fix:** Use parameterized Supabase filters:
```typescript
const query = supabase.from("patients").select("...");
if (email) query.eq("email", email);
if (document) query.eq("document", document);
// Or use proper .or() with filter builder methods
```

### SEC-02: Missing "use server" Directive in consultationService.ts [CRITICAL]

**File:** `src/services/consultationService.ts:1`

Unlike `medicService.ts` and `patientsService.ts`, this file does **not** have a `"use server"` directive at the top. This means Next.js may bundle it on the client side. Since it imports `CreateServerClient()` (which uses `SUPABASE_SERVICE_ROLE_KEY`), this could expose the service role key in client bundles.

**Impact:** Exposure of the Supabase service role key, which bypasses all Row Level Security. Full database access.

**Fix:** Add `"use server";` as the first line of the file.

---

### SEC-03: Sign Out Does Not Invalidate Session [HIGH]

**File:** `src/components/Header.tsx:62-65`

```tsx
<Link href="/login" className="...">Sign Out</Link>
```

The "Sign Out" button is a plain link to `/login`. It does **not** call `signOut()` from `next-auth/react`, meaning the JWT session remains valid. A user who "signs out" is still authenticated, and anyone with access to the browser can navigate back to protected pages.

**Fix:** Replace with a button that calls `signOut({ callbackUrl: "/login" })`.

---

### SEC-04: Incomplete Middleware Route Protection [HIGH]

**File:** `middleware.ts:36-38`

```typescript
matcher: ["/dashboard/:path*", "/login", "/register"],
```

The middleware matcher only protects `/dashboard/:path*`. The following authenticated routes are **unprotected** by middleware:
- `/profile`
- `/patients/lookup`
- `/patients/profile`
- `/consultation`
- `/settings`

**Impact:** Any unauthenticated user can access patient data, consultation forms, and medic profiles by directly navigating to these URLs.

**Fix:** Expand the matcher:
```typescript
matcher: ["/dashboard/:path*", "/profile/:path*", "/patients/:path*", "/consultation/:path*", "/settings/:path*", "/login", "/register"],
```

---

### SEC-05: No Authorization in Server Actions [HIGH]

**Files:** All files in `src/services/`

Server actions like `getMedicData(email)`, `getPatientById(id)`, `saveMedicProfile(email, profile)`, and `patientLookupQuery(email, document)` accept identifiers as parameters but never verify that the **calling user** is authorized to access or modify the specified records. Any authenticated user can read or modify any other user's data by calling these server actions with arbitrary IDs/emails.

**Fix:** Each server action should validate the caller's session and verify they have permission to access the requested resource.

---

### SEC-06: Weak Password Policy [MEDIUM]

**File:** `src/lib/validators.tsx:22-38`

The password validator only enforces a minimum length of 6 characters with no complexity requirements (uppercase, lowercase, numbers, special characters). For a medical application handling PHI (Protected Health Information), this does not meet healthcare security standards (e.g., HIPAA).

**Fix:** Enforce at least 8 characters with complexity requirements, or integrate a password strength library.

## Bugs

### BUG-01: `createNewPatient` Inserts 14 Separate Rows Instead of 1 [CRITICAL]

**File:** `src/services/patientsService.ts:67-84`

```typescript
const { error, data } = await supabase
  .from("patients")
  .insert([
    { email: patient.email },
    { first_name: patient.first_name },
    { last_name: patient.last_name },
    // ... each field as a separate object
  ]);
```

Supabase's `.insert()` treats each array element as a **separate row**. This code attempts to insert 14 rows, each with only one column populated. This will either fail due to NOT NULL constraints or create 14 garbage rows.

**Fix:**
```typescript
const { error, data } = await supabase
  .from("patients")
  .insert({
    email: patient.email,
    first_name: patient.first_name,
    last_name: patient.last_name,
    // ... all fields in one object
  });
```

### BUG-03: Missing `await` on Registration [CRITICAL]

**File:** `src/app/register/page.tsx:52`

```typescript
const data = signMedic(email, password); // Missing await!
console.log("Registry result: ", data);
router.push("/login?registered=true");
```

`signMedic()` is an async server action. Without `await`, the Promise is never resolved, errors are never caught by the `try/catch`, and the user is always redirected to login regardless of whether registration succeeded or failed.

**Fix:** Add `await`: `const data = await signMedic(email, password);`

### BUG-04: Profile Page Birth Date Field Name Mismatch [HIGH]

**File:** `src/app/(loggedPages)/profile/page.tsx:199-206`

```tsx
<input
  id="birth_year"
  name="birth_year"   // <-- name is "birth_year"
  type="date"
  value={profile.birth_date || ""}  // <-- reads from "birth_date"
  onChange={handleChange}  // <-- sets profile["birth_year"]
/>
```

The input `name` is `"birth_year"` but the profile state key is `"birth_date"`. The `handleChange` function uses `e.target.name` as the key, so it writes to `profile.birth_year` (which doesn't exist in the type), while the value reads from `profile.birth_date`. The date picker will appear to work but changes are never persisted correctly.

**Fix:** Change `name="birth_year"` to `name="birth_date"`.

### BUG-05: useEffect Without Dependency Array Causes Infinite Loop [HIGH]

**File:** `src/app/(loggedPages)/consultation/page.tsx:19-21`

```typescript
useEffect(() => {

});
```

This `useEffect` has **no dependency array**, meaning it runs after **every render**. While the body is currently empty, this is a ticking time bomb - any code added to it will cause infinite re-renders. It should either have a dependency array or be removed.

**Fix:** Remove the empty useEffect, or add `[]` if it's meant for initialization.

### BUG-07: Patient Profile Formik Missing `enableReinitialize` [MEDIUM]

**File:** `src/app/(loggedPages)/patients/profile/page.tsx:101-118`

Formik's `initialValues` are set from `patientProfile` state, which starts empty and is populated asynchronously via `getPatientById`. Since Formik captures initial values at mount time and doesn't re-read them when state changes, the form will always show empty fields even after the data loads.

## Code Quality & Inconsistencies

### CQ-01: Duplicate Type Definitions [HIGH]

- `src/services/medicService.ts` exports `Profile` type
- `src/app/(loggedPages)/profile/page.tsx:12-20` defines an identical local `Profile` type
- `src/services/patientsService.ts:48-63` defines `PatientData` (snake_case)
- `src/lib/types/patientTypes.ts` defines `PatientProfile` (camelCase)

The same entities are defined in multiple places with different naming. This leads to type drift and maintenance burden.

**Recommendation:** Centralize all types in `src/lib/types/` and import from there.

---

### CQ-02: Inconsistent Error Handling Across Services [HIGH]

| Service Function | On Error |
|---|---|
| `signMedic` | `throw error` |
| `getMedicData` | returns `null`, logs `console.error` |
| `saveMedicProfile` | `throw error` |
| `patientLookupQuery` | returns `null`, logs `console.error` |
| `getPatientById` | returns `null`, logs `console.error` |
| `createNewPatient` | returns `null`, logs `console.error` |
| `createConsultation` | `throw error` |
| `listConsultation` | `throw error` |
| `getConsultationByMedic` | catches and logs `console.log` |
| `getConsultationByPatient` | catches and logs `console.log` |
| `getConsultationById` | `throw error` |

This inconsistency makes error handling unpredictable for callers. Some throw (requiring try/catch), some return null (requiring null checks), and some silently swallow errors.

**Recommendation:** Adopt a consistent pattern. Consider using a `Result<T, E>` type or always throw and let callers handle errors.

### CQ-03: Barrel Export Does Not Capture Default Exports [MEDIUM]

**File:** `src/components/index.ts`

```typescript
export * from "./Sidebar";
export * from "./MainLayout";
export * from "./HeaderCard";
```

Components like `Sidebar`, `MainLayout`, `HeaderCard`, and `QueryResult` use `export default`, which is **not** re-exported by `export *`. Only named exports (`Button`, `FormikInputField`, `FormikTextAreaField`, `InputField`) are available through the barrel file.

**Recommendation:** Use named exports consistently, or add explicit re-exports: `export { default as Sidebar } from "./Sidebar"`.

## Improvement Opportunities

### IMP-01: Add Server-Side Input Validation

All server actions accept input parameters without validation. Add Zod or similar runtime schema validation at the service boundary to ensure data integrity.

### IMP-02: Implement Rate Limiting

Auth endpoints (`signIn`, `signUp`) have no rate limiting. Implement rate limiting to prevent brute-force attacks, especially critical for a medical application.

### IMP-03: Add CSRF Protection

Ensure CSRF tokens are validated on form submissions, particularly for state-changing operations like profile updates and patient creation.

### IMP-04: Add Loading Skeletons

Replace plain "Loading..." text with skeleton UI components for better perceived performance.

### IMP-05: Implement Proper Error Boundaries

Add React Error Boundaries to gracefully handle component-level errors instead of crashing the entire page.

### IMP-06: Add Audit Logging

For a medical application, all data access and modifications should be logged for compliance purposes (who accessed what patient data, when).

### IMP-07: Implement Proper Session Management

The JWT session has a 30-day maxAge with no refresh logic. Consider shorter session durations with refresh tokens for better security.
